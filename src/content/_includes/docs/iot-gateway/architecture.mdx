## Connector

The purpose of this component is to connect to external systems (like MQTT brokers or OPC-UA servers) or directly to devices (like Modbus, BLE, or CAN). Once connected, the connector either polls data from those systems or subscribes to updates. Whether to poll or subscribe depends on the protocol capabilities. For example, MQTT connectors use a subscription model, while Modbus and CAN use polling. The connector is also able to push updates to devices either directly or via external systems.

You can define your own connector using the <DocLink product={props.product} path='custom'>customization guide</DocLink>.

## Converter

Converters are responsible for converting data from a protocol-specific format to and from the ThingsBoard format. Converters are invoked by Connectors and are often specific to the protocol supported by the Connector. There are uplink and downlink converters. The uplink converter converts data from a specific protocol to ThingsBoard format. The downlink converter converts messages from ThingsBoard to a specific protocol format.

You can define your own converter using the <DocLink product={props.product} path='custom#step-4-define-converter-implementation'>customization guide</DocLink>.

## Event Storage

The Event Storage temporarily stores telemetry and other events produced by Connectors until they are delivered to ThingsBoard. It supports two implementations: in-memory queue and persistent file storage. Both implementations ensure that device data is eventually delivered in case of network outages. The in-memory queue minimizes IO operations but may lose messages if the gateway process restarts. Persistent file storage survives process restarts but performs IO operations to the file system.

## ThingsBoard Client

The Gateway communicates with ThingsBoard via MQTT protocol and uses the API described in the Gateway MQTT API reference. ThingsBoard Client is a separate thread that polls Event Storage and delivers messages when the connection to ThingsBoard is active. It supports connectivity monitoring, event batching for performance, and other features.

## Gateway Service

The Gateway Service is responsible for bootstrapping the Connectors, Event Storage, and ThingsBoard Client. It collects and periodically reports statistics to ThingsBoard about incoming messages and connected devices. Gateway Service persists the list of connected devices to re-subscribe to device configuration updates after a gateway restart.
