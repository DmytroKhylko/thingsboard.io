The **Notification center** is where you send, manage, and automate notifications. You can trigger them manually, via the REST API, or automatically based on system events, with optional scheduling.

```plaintext
  trigger event                       ┌─────────┐
  (alarm, device activity, etc.)      │ Template │  ← subject, message, params
        │                             └────┬────┘
        ▼                                  │
  ┌───────────┐   matches    ┌─────────────▼──────────────┐
  │   Rule    │ ──────────▶  │ Resolve recipients & methods│
  └───────────┘              └─────────────┬──────────────┘
                                           │
                    ┌──────────────────────┬┴────────────────────┐
                    ▼                      ▼                     ▼
              ┌──────────┐          ┌──────────┐          ┌──────────┐
              │   Web    │          │  Email   │          │  Slack   │  ...
              └──────────┘          └──────────┘          └──────────┘
```

## Notification delivery methods

- **Web:** Notifications appear inside the ThingsBoard web interface.
- **Mobile app:** Push notifications via the {props.product === Products.PE ? <DocLink product={props.product} path="pe/mobile">ThingsBoard PE Mobile Application</DocLink> : <DocLink product={props.product} path="mobile">ThingsBoard Mobile Application</DocLink>}. Requires mobile app setup and mobile settings enabled in ThingsBoard.
- **SMS:** Delivered via SMS. Works without internet access. Requires an <DocLink product={props.product} path="user-guide/ui/sms-provider-settings">SMS provider</DocLink>{props.product === Products.CE ? <> configured by a System Administrator</> : <> configured at your level or by a System Administrator</>}.
- **Email:** Delivered to the recipient's inbox. Requires an <DocLink product={props.product} path="user-guide/ui/mail-settings">outgoing mail server</DocLink>{props.product === Products.CE ? <> configured by a System Administrator</> : <> configured at your level or by a System Administrator</>}.
- **Slack:** Messages to Slack users or channels. Requires <DocLink product={props.product} path="user-guide/ui/slack-settings">Slack settings</DocLink>.
- **Microsoft Teams:** Messages to a Teams channel. Requires a <DocLink product={props.product} path="user-guide/ui/microsoft-teams-settings">webhook URL</DocLink>.

## Managing notifications

**Inbox:** View incoming notifications in the **Inbox** tab. By default, only unread messages appear. Click **All** to see all. Mark as read or delete from this view, or use the bell icon in the top-right corner for quick access.

## Recipients

The **Recipients** tab lists configured notification recipients. To add one, click **Add recipients**, enter a name, select a type, fill in the type-specific fields, and click **Add**. You can also create recipients inline during the send wizard.

- **Platform users:** ThingsBoard users. Choose a user filter: **All users**, **Tenant administrators**, **User list**, **Customer users**, **Users of the entity owner**, or **Affected user**{props.product !== Products.CE && <>, **User group list**, or **User role**</>}.

- **Microsoft Teams:** A Teams channel. Requires a <DocLink product={props.product} path="user-guide/ui/microsoft-teams-settings">webhook URL</DocLink>. Enter the webhook URL and channel name.

## Templates

The **Templates** tab manages notification templates. Each template defines a set of delivery methods and message content per method.

Click **Add template**, enter a name, select a type and delivery methods, enter the subject and message for each method, and click **Add**.

Templates support dynamic parameters wrapped in `${...}`, like `${recipientFirstName}`, with optional suffixes (`upperCase`, `lowerCase`, `capitalize`). See <DocLink product={props.product} path="reference/notification-system/template-parameters">Notification template parameters</DocLink> for the full reference.

## Rules

Notification rules automate delivery based on trigger events. Each rule is coupled to a specific template type.

<Steps>

1. Go to the **Rules** tab and click **Add rule**.

2. Enter a name and select a trigger type.

3. Select a template or [create a new one](#templates).

4. Specify recipients and click **Next**.

5. Configure the trigger filter and click **Add**.

</Steps>

### Alarm escalation chain

The **Alarm** trigger uses an escalation chain instead of a simple recipient list. You configure multiple recipient stages with delays between them:

```plaintext
  alarm created
       │
       ▼
  ┌────────────────┐                    ┌────────────────┐
  │ First recipient│  delay (e.g. 5m)   │ Second recipient│  delay
  │ (immediate)    │ ─────────────────▶  │                │ ─────▶ ...
  └────────────────┘                    └────────────────┘
       │                                      │
       │  alarm cleared / acknowledged        │
       └──────── escalation stops ────────────┘
```

If a recipient acknowledges or clears the alarm before the next stage's delay expires, subsequent recipients do not receive the notification. Configure which alarm statuses stop the chain in the **Stop escalation on the alarm status become** field.
