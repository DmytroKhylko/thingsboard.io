The **Notification center** is where you send, manage, and automate notifications. You can trigger them manually, via the REST API, or automatically based on system events, with optional scheduling.

## Notification delivery methods

- **Web:** Notifications appear inside the ThingsBoard web interface.
- **Mobile app:** Push notifications via the {props.product === Products.PE ? <DocLink product={props.product} path="pe/mobile">ThingsBoard PE Mobile Application</DocLink> : <DocLink product={props.product} path="mobile">ThingsBoard Mobile Application</DocLink>}. Requires mobile app setup and mobile settings enabled in ThingsBoard.
- **SMS:** Delivered via SMS. Works without internet access. Requires an <DocLink product={props.product} path="user-guide/ui/sms-provider-settings">SMS provider</DocLink>{props.product === Products.CE ? <> configured by a System Administrator</> : <> configured at your level or by a System Administrator</>}.
- **Email:** Delivered to the recipient's inbox. Requires an <DocLink product={props.product} path="user-guide/ui/mail-settings">outgoing mail server</DocLink>{props.product === Products.CE ? <> configured by a System Administrator</> : <> configured at your level or by a System Administrator</>}.
- **Slack:** Messages to Slack users or channels. Requires <DocLink product={props.product} path="user-guide/ui/slack-settings">Slack settings</DocLink>.
- **Microsoft Teams:** Messages to a Teams channel. Requires a <DocLink product={props.product} path="user-guide/ui/microsoft-teams-settings">webhook URL</DocLink>.

## Managing notifications

**Inbox:** View incoming notifications in the **Inbox** tab. By default, only unread messages appear. Click **All** to see all. Mark as read or delete from this view, or use the bell icon in the top-right corner.

**Sent:** The **Sent** tab lists all notifications you have sent and their delivery status. Resend by clicking **Notify again** (+) or delete using the delete icon.

<Aside type="caution">
Deleting a Web notification removes it for all recipients. Delivery issues appear in the **Status** column.
</Aside>

## Recipients

The **Recipients** tab lists configured notification recipients. To add one, click **Add recipients**, enter a name, select a type, fill in the type-specific fields, and click **Add**. You can also create recipients inline during the send wizard.

- **Platform users:** ThingsBoard users. Choose a user filter: **All users**, **Tenant administrators**, **User list**, **Customer users**, **Users of the entity owner**, or **Affected user**{props.product !== Products.CE && <>, **User group list**, or **User role**</>}.
- **Slack:** A Slack channel or direct message. Requires <DocLink product={props.product} path="user-guide/ui/slack-settings">Slack settings</DocLink> configured first. Select the channel type and specify the conversation.
- **Microsoft Teams:** A Teams channel. Requires a <DocLink product={props.product} path="user-guide/ui/microsoft-teams-settings">webhook URL</DocLink>. Enter the webhook URL and channel name.

## Templates

The **Templates** tab manages notification templates. Each template defines a set of delivery methods and message content per method.

Click **Add template**, enter a name, select a type and delivery methods, enter the subject and message for each method, and click **Add**.

Templates support dynamic parameters wrapped in `${...}`, like `${recipientFirstName}`, with optional suffixes (`upperCase`, `lowerCase`, `capitalize`). See <DocLink product={props.product} path="user-guide/notification-template-parameters">Notification template parameters</DocLink> for the full reference.

## Send notification

<Steps>

1. Click **Send notification** in the top-right corner.

2. Select recipients and delivery methods. Optionally schedule the notification{props.product !== Products.CE && <> or <DocLink product={props.product} path="user-guide/reporting/attaching-report-to-notification">attach a report</DocLink> (Email and Slack only)</>}. Click **Next**.

3. Enter the subject and message for each delivery method. For Web notifications, optionally add an action button with a dashboard link or URL. Click **Next**.

4. Review and click **Send**.

</Steps>

## Rules

Notification rules automate delivery based on trigger events. Each rule is coupled to a specific template type.

<Steps>

1. Go to **Rules** and click **Add rule**.

2. Enter a name and select a trigger type.

3. Select a template or [create a new one](#templates).

4. Specify recipients and click **Next**.

5. Configure the trigger filter and click **Add**.

</Steps>

<Aside type="note">
The **Alarm** trigger uses an escalation chain in step 4 instead of a simple recipient list. Add recipient groups, configure delays between them, and set which alarm statuses stop the chain. If the first recipient clears the alarm before a later group's delay expires, subsequent recipients do not receive the notification.
</Aside>

| Trigger | When it fires | Trigger filter | Default rule |
|---------|---------------|----------------|--------------|
| **Alarm** | Alarm created, acknowledged, cleared, severity changed, or deleted | Type, severity, event types | *New alarm* |
| **Device activity** | Device becomes active or inactive | Devices or device types, event types | *Device activity status change* |
| **Entity action** | Entity created, updated, or deleted | Entity type, action types | *Device created* |
| **Alarm comment** | Comment added or updated on an alarm | Type, severity, status; optionally user comments only | *Comment on active alarm* |
| **Alarm assignment** | Alarm assigned or unassigned | Type, severity, status, event types | *Alarm assignment* |
| **Rule engine lifecycle event** | Rule chain or node lifecycle event | Chain or node, status | *Rule node initialization failure* |
| **Edge connection** | Edge connection status changes | Edge instances, event types | *Edge connection status change* |
| **Edge communication failure** | Edge communication failure | Edge instances | *Edge communication failure* |
| **Entities limit** | Tenant approaching entity count limit | Entity type, threshold % | *Entities Limit* |
| **API usage limit** | Tenant reaching API usage threshold | API feature, event types | *API usage limit* |
| **New platform version** | New ThingsBoard version released | None | *New platform version* |
| **Exceeded rate limits** | Rate limit exceeded | API feature | *Exceeded rate limits* |
